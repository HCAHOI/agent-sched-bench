## Executive Summary

The project has **all required modules implemented as standalone units**, but suffers from **critical integration gaps** that would prevent a successful end-to-end experiment run. The previous Gemini CLI review (2026-04-01) marked everything 🟢 without verifying cross-module integration — this report corrects that assessment.

**Verdict: NOT READY for experiment runs.** Five critical issues must be resolved first.

## 🔴 Critical Issues (Must fix before any experiment)

### C-1: Runner ↔ MetricsCollector not integrated
- **Files**: `src/harness/runner.py`, `src/harness/sweep.py`
- **Problem**: `VLLMMetricsCollector` exists but is **never called** during benchmark runs. Neither `BenchmarkRunner.run()` nor `execute_sweep()` start a concurrent metrics polling task. This means **Group A metrics** (GPU util, KV cache usage, preemption counts) will NOT be collected during experiments.
- **Spec requirement**: Section 4 — "每秒能采集到 GPU util + vLLM metrics"; HARNESS-2 — metrics must run concurrently with benchmark.
- **Impact**: Without concurrent metrics, there is no KV cache utilization time series, no preemption count tracking, and no GPU utilization data. The core research question ("why does throughput drop at high N?") cannot be answered.
- **Fix**: Add a `metrics_collector` parameter to `BenchmarkRunner` or `execute_sweep()`, start `collector.poll()` as a concurrent `asyncio.Task`, cancel it when the run completes.

### C-2: Runner ↔ TraceLogger not integrated
- **Files**: `src/harness/runner.py`, `src/harness/trace_logger.py`
- **Problem**: `TraceLogger` is implemented with `log_step()` and `log_summary()` methods, but `BenchmarkRunner` never instantiates or calls it. The runner accumulates results in memory and returns them. `sweep.py` writes results as monolithic JSON files per run cell, NOT as JSONL.
- **Spec requirement**: HARNESS-3 — "每个 StepRecord 都有精确的 timestamp", JSONL format for streaming analysis.
- **Impact**: (1) No real-time JSONL trace output during long runs — if a run crashes mid-way, partial traces are lost. (2) `TraceReplayer` expects JSONL with `type` and `program_id` fields, but the actual sweep output is nested JSON. Traces can't be replayed without a manual conversion step that doesn't exist.
- **Fix**: Integrate `TraceLogger` into `BenchmarkRunner._run_single_task()` to write each `StepRecord` as it completes.

### C-3: Sweep matrix missing required system configs
- **File**: `configs/sweep.yaml`
- **Problem**: Matrix only includes `vllm-baseline`, `continuum`, `thunderagent`. Missing `vllm-preserve` and `vllm-no-preempt`, which both have config files in `configs/systems/`.
- **Spec requirement**: ENV-5 — "实验需要的三种 vLLM 配置" (vllm-baseline, vllm-preserve, vllm-no-preempt). Section 9, Decision #6 — "必须控制 vLLM preemption".
- **Impact**: Cannot reproduce the "preserve always wins because of preemption safety valve" confound analysis. Cannot generate the motivation figure showing preemption as confound. The entire benchmark's raison d'etre depends on comparing these configs.
- **Fix**: Add `vllm-preserve` and `vllm-no-preempt` to `configs/sweep.yaml` matrix systems.

### C-4: No data files exist
- **File**: `data/` (only contains `README.md`)
- **Problem**: All workload configs point to `data/swebench_lite`, `data/bird_sql`, `data/research_queries` — none of which exist. `sweep.py` will crash at `resolve_task_source()` with `FileNotFoundError`.
- **Impact**: Cannot run any experiment. This is a prerequisite for everything else.
- **Fix**: Either download real datasets or create representative task JSON files for smoke testing. The spec's Phase 1 Day 2 requires at least SWE-bench tasks to be available.

### C-5: Test suite fails
- **File**: `tests/test_analysis.py`, `pyproject.toml`
- **Problem**: `matplotlib` is imported by `analysis/plots.py` but not listed in `pyproject.toml` dependencies. `pytest` fails with `ModuleNotFoundError: No module named 'matplotlib'`.
- **Impact**: CI/test gate broken. Cannot verify analysis module.
- **Fix**: Add `matplotlib>=3.8,<4.0` to `[project.dependencies]` in `pyproject.toml`.

## 🟠 Major Issues (Should fix before experiments)

### M-1: `lifecycle_mode` / `preserve_strategy` configs not implemented
- **Files**: `configs/systems/vllm_preserve.yaml`, `src/serving/engine_launcher.py`
- **Problem**: `vllm_preserve.yaml` defines `lifecycle_mode: preserve` and `preserve_strategy: keep_alive`, but no code in `engine_launcher.py` or elsewhere reads or acts on these fields. The "preserve" behavior (keep KV cache alive between turns) has no implementation.
- **Impact**: Even if `vllm-preserve` is added to the sweep, it would behave identically to `vllm-baseline` — the config fields are dead.

### M-2: Metrics collector missing histogram metrics
- **Files**: `src/harness/metrics.py`
- **Problem**: Spec lists `vllm:e2e_request_latency_seconds` and `vllm:time_to_first_token_seconds` in METRICS_OF_INTEREST. These are Prometheus histograms (with `_bucket`, `_sum`, `_count` suffixes). The current `_parse_prometheus()` method only handles simple `metric_name value` lines and cannot parse histogram format.
- **Impact**: Missing TTFT and E2E latency distribution data. These are key diagnostic metrics for understanding scheduling behavior.

### M-3: plots.py cannot produce multi-system comparison plots
- **Files**: `src/analysis/plots.py`
- **Problem**: All plot functions take a single DataFrame and render one line/bar series. The spec requires overlay plots comparing systems: "Throughput vs N 曲线 (per system, per workload)". Also, `main()` only calls `plot_throughput_vs_concurrency` — `plot_latency_breakdown` and `plot_prefix_cache_hit_vs_concurrency` are unreachable from CLI.
- **Impact**: Cannot generate the key paper figures comparing system performance.

### M-4: REPLACE_WITH_COMMIT_OR_TAG placeholders still present
- **Files**: `configs/systems/continuum.yaml:7`, `configs/systems/thunderagent.yaml:7`, `scripts/serve_continuum.sh:8`, `scripts/serve_thunderagent.sh:8`
- **Problem**: Git refs for Continuum and ThunderAgent are placeholder strings. Running experiments with these would clone HEAD (unstable), violating reproducibility requirements.
- **Impact**: Non-reproducible experiments. The spec explicitly requires "接受标准必须绑定 immutable ref".

### M-5: `vllm_no_preempt.yaml` has `max_num_seqs: 1`
- **File**: `configs/systems/vllm_no_preempt.yaml:17`
- **Problem**: Setting `max_num_seqs=1` serializes ALL requests — only one can run at a time. This doesn't "disable preemption"; it eliminates concurrency entirely. With N > 1 agents, requests queue sequentially, which makes the benchmark measure queueing delay, not scheduling efficiency.
- **Impact**: Benchmark results with this config would be meaningless for the research question. The intent is to study what happens when preemption is disabled under contention, not to eliminate contention.
- **Fix**: Need a different approach — either a vLLM source patch, scheduler hook, or a much higher `max_num_seqs` with a mechanism to prevent eviction.

### M-6: Previous review was misleadingly positive
- **File**: `docs/IMPLEMENTATION_REVIEW.md`
- **Problem**: The Gemini CLI review marked all components 🟢 Complete and concluded "READY FOR RUNTIME ACCEPTANCE". It did not catch any of the critical integration issues (C-1 through C-5). It verified component existence but not component integration.
- **Impact**: False confidence in readiness. This is exactly the "tunnel vision" scenario the review gate was designed to prevent.
- **Recommendation**: Treat the Gemini review as a component checklist only, not an integration assessment.

### M-7: `parse_traces.py` only computes 3 of 7+ Group A metrics
- **File**: `src/analysis/parse_traces.py`
- **Problem**: `summarize_trace_frame()` computes `throughput_steps_per_min`, `avg_jct_s`, and `p90/p95_jct_s`. Missing: GPU Utilization (from nvidia-smi), KV Cache Usage (from vLLM metrics), and these can't be added without C-1 being fixed first.
- **Impact**: Analysis pipeline produces incomplete diagnostic output.

## 🟡 Minor Issues

### N-1: No progress reporting in runner/sweep
- **Spec**: HARNESS-1 TODO — "添加 progress bar (tqdm)"
- **Status**: Not implemented. No `tqdm` in dependencies. Long sweep runs (63 cells in full matrix) will show no progress indication.

### N-2: `nvidia-smi` format mismatch
- **File**: `src/harness/metrics.py:96-103`
- **Problem**: `sample_nvidia_smi()` calls `nvidia-smi` with `--format=csv` but the `parse_nvidia_smi_csv()` parser expects no header / no units. The CSV output with headers will include "utilization.gpu [%], memory.used [MiB]" as the first row, which the parser tries to skip via string matching but could be fragile across nvidia-smi versions.

### N-3: Smoke workload configs unused
- **Files**: `configs/workloads/code_agent_smoke.yaml`, `data_agent_smoke.yaml`, `research_agent_smoke.yaml`
- **Problem**: These exist but are not referenced by `sweep.yaml`, `Makefile`, or any script. The `run_smoke.sh` / `Makefile smoke-*` targets run pytest, not the benchmark harness with smoke configs.

### N-4: `vllm_preserve.yaml` missing health section
- **File**: `configs/systems/vllm_preserve.yaml`
- **Problem**: Unlike `vllm_baseline.yaml` which has `health.timeout_s` and `health.poll_interval_s`, this config has no health section. Health check code may fail or use hardcoded defaults.

### N-5: Makefile doesn't have `make build` target
- **File**: `Makefile`
- **Problem**: Project memory says `build=make build` but only `make sync` exists for dependency installation.

### N-6: Sweep system name mapping is fragile
- **Files**: `configs/sweep.yaml`, `src/harness/sweep.py:166`
- **Problem**: Sweep uses `vllm-baseline` (dash) but config file is `vllm_baseline.yaml` (underscore). Code does `run.system.replace('-', '_')` to bridge this. Easy to break when adding new systems.

## Additional Findings from Parallel Review Agents

Three independent code-reviewer agents examined the agents, harness, and serving+analysis modules respectively. Key additional findings not covered above:

### From Agents Module Review

| ID | Severity | File:Line | Issue |
|:---|:---|:---|:---|
| A-1 | 🔴 Critical | `data_agent.py:69` | **SQL injection**: table name in `schema_inspect` is f-string interpolated into `PRAGMA table_info({table})` without sanitization. A malicious task payload could execute arbitrary SQL. |
| A-2 | 🟠 Major | `base.py:96` | **Unguarded `choices[0]`**: `_call_llm()` accesses `response.choices[0]` without checking if `choices` is empty. An API error or rate-limit response with no choices will crash with `IndexError`. |
| A-3 | 🟠 Major | `research_agent.py:79` | **Missing `synthesize` tool**: spec lists three tools (`web_search`, `read_page`, `synthesize`) but implementation only supports `web_search` and `page_read`. Also `read_page` (spec) vs `page_read` (code) naming mismatch. |
| A-4 | 🟡 Minor | `code_agent.py:194`, `data_agent.py:184`, `research_agent.py:183` | **No tool output truncation**: tool results are appended to messages without length limits. A bash command returning megabytes or a large web page will blow out the LLM context window. Should truncate to configurable max (e.g., 8000 chars). |
| A-5 | 🟡 Minor | `data_agent.py:120-121` | **Unguarded `json.loads`**: parsing final SQL result can raise `JSONDecodeError` if result is malformed despite passing the `startswith("ERROR:")` check. |
